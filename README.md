# VPN Site Manager

Интерактивный установщик и менеджер production-развёртывания сайта `dorillo/vpn-site` на чистой Ubuntu 24.04. Скрипт устанавливает системные зависимости, собирает Python 3.14, настраивает PostgreSQL, systemd, Nginx, Let's Encrypt, резервное копирование и при необходимости UFW.

Поддерживаются только Ubuntu 24.04 на `amd64` и `arm64`. Для установки требуются не менее 1 ГиБ RAM и 4 ГиБ свободного места. Скрипт рассчитан на отдельный сервер без другого сайта на портах 80 и 443.

## Перед установкой

Подготовьте:

- домен с публичной записью A и/или AAAA, указывающей на сервер; canonical,
  `robots.txt` и `sitemap.xml` выбранного frontend release должны быть
  подготовлены для этого же домена;
- открытые извне TCP-порты 80 и 443;
- SMTP-сервер для кодов входа;
- URL и токен Remnawave, а также cookie reverse-proxy, если она требуется;
- email первого администратора;
- fine-grained GitHub token с доступом только к приватному репозиторию сайта и разрешением `Contents: Read-only`.

Если включаете UFW, заранее выясните фактический TCP-порт SSH. Ошибочный порт может закрыть удалённый доступ. YooKassa при первой установке необязательна.

## Установка

Запустите на сервере от пользователя с `sudo`:

```bash
curl -fsSL https://raw.githubusercontent.com/dorillo/Website-management-and-installation-script/main/install.sh | sudo bash -s -- install
```

Установщик обновит пакеты Ubuntu и запросит домен, SMTP, Remnawave, email первого администратора и при необходимости YooKassa/UFW. Внешний вид, публичные тексты, логотип и ИНН теперь входят в сам frontend release и через env не настраиваются. `SITE_NAME` считывается из `.env.example` выбранного release и остаётся backend-настройкой для писем, описаний платежей и API. Сборка Python 3.14 из исходников может занять несколько минут. Перед открытием HTTPS наружу менеджер запускает приложение с отключёнными background jobs за локальным Nginx, проверяет миграции, startup prerequisites, health и матрицу маршрутов. После запуска production установщик попросит войти с email первого администратора и подтвердить получение прав. Только после подтверждения `ADMIN_BOOTSTRAP_EMAILS` очищается, а сайт перезапускается. Если отложить подтверждение, завершить процедуру можно позже в настройках окружения. После установки команда управления доступна как:

```bash
sudo vpn-site
```

GitHub-токен используется только в памяти текущего процесса и очищается после операции. Он не записывается в env-файл, конфигурацию менеджера, URL команды или логи. При каждом обновлении приватного сайта токен запрашивается снова.

В блоке Remnawave установщик также предложит cookie reverse-proxy. Правильнее всего вводить её без фигурных скобок и кавычек: `aEmFnBcC=WbYWpixX`. Если значение скопировано из Nginx как `"~*aEmFnBcC=WbYWpixX"`, его тоже можно вставить целиком: менеджер удалит служебный префикс `~*`. Старый JSON-формат `{"aEmFnBcC":"WbYWpixX"}` по-прежнему поддерживается. Формат `{"aEmFnBcC=WbYWpixX"}` не является корректным JSON, но менеджер принимает и нормализует его для совместимости. Ввод скрыт. Нажатие Enter при первой установке записывает `{}`; при изменении настроек оставляет текущее значение.

## Команды

| Команда | Назначение |
| --- | --- |
| `sudo vpn-site` | Интерактивное меню |
| `sudo vpn-site install` | Мастер первоначальной установки |
| `sudo vpn-site status` | Состояние сервисов и локальная health-проверка |
| `sudo vpn-site start` | Запуск backend |
| `sudo vpn-site stop` | Остановка backend; Nginx остаётся запущен |
| `sudo vpn-site restart` | Перезапуск с health-проверкой |
| `sudo vpn-site update` | Обновление сайта из приватного GitHub-репозитория |
| `sudo vpn-site backup` | Неинтерактивный бэкап PostgreSQL и конфигурации |
| `sudo vpn-site diagnose` | Проверка сервисов, env, HTTPS, сертификата и таймера бэкапов |
| `sudo vpn-site logs` | Последние 200 строк журнала приложения и ошибки Nginx |
| `sudo vpn-site update-manager` | Обновление менеджера из публичного репозитория |
| `sudo vpn-site os-update` | Обновление пакетов Ubuntu |
| `sudo vpn-site repair` | Восстановление управляемых файлов systemd, Nginx и Certbot |

Команды `status`, `start`, `stop`, `restart`, `backup`, `diagnose`, `logs`, `os-update` и `repair` можно вызывать без интерактивного терминала, например из системы мониторинга. Установка, меню и обновление приватного сайта требуют TTY.

## Возможности меню

В разделе настроек окружения можно менять SMTP, Remnawave, YooKassa, rate limits и параметры фоновой отправки уведомлений, а также завершить выдачу прав первого администратора. При завершении `ADMIN_BOOTSTRAP_EMAILS` очищается только после подтверждения успешного входа, затем сайт перезапускается с health-проверкой. Полный env-файл также можно открыть в `${SUDO_EDITOR:-${EDITOR:-nano}}`; перед применением создаётся резервная копия, конфигурация валидируется кодом сайта, а сервис проходит health-проверку. При ошибке или прерывании восстанавливается предыдущий файл.

Раздел сертификатов показывает срок действия и выполняет тестовое или принудительное продление. Автоматическая смена домена сохраняется только для legacy release с runtime-конфигурацией frontend. В новой статической версии canonical, Open Graph URL, `robots.txt` и `sitemap.xml` входят в release, поэтому менеджер не переписывает их и отказывает в неполном переносе домена. Сертификаты выдаются через Let's Encrypt webroot. Автопродление выполняет системный `certbot.timer`, после успешного продления Nginx проверяется и перезагружается.

Раздел бэкапов создаёт, показывает и восстанавливает дампы базы, а также позволяет изменить срок хранения от 1 до 3650 дней. По умолчанию бэкапы хранятся 14 дней.

## Обновления и откат

Каждая версия сайта разворачивается в отдельный каталог. Backend сначала останавливается, чтобы после authoritative custom-format дампа PostgreSQL не осталось окна для потерянных записей. Затем одной update-транзакцией применяются актуальная env-схема, миграции Alembic, systemd, Nginx и release. До commit новая версия проверяется с отключённым lifespan за Nginx, разрешающим доступ только с localhost: production-трафик и фоновые YooKassa, SMTP и Remnawave-операции в rollback-окно не запускаются. Ошибка или прерывание до commit возвращает прежнюю базу, env, runtime-файлы, release, Git ref, состояние backup timer и исходное состояние backend. Настоящий lifespan запускается только после commit; если этот редкий финальный запуск не проходит health-check, новая БД сохраняется, а публичный доступ остаётся закрытым, чтобы не откатывать уже возможные внешние эффекты.

Переход через миграцию сайта `20260720_0006` безопасно автоматизируется только для базы без legacy-платежей. Текущий `vpn-site` сначала помечает всю старую историю как `auto_capture=false`, а затем намеренно запрещает удалить этот признак. Менеджер обнаруживает такую базу до изменения данных и останавливает обновление: нельзя удалять финансовую историю, подменять `auto_capture=true` или принудительно stamp-ить Alembic. Для production-базы с платежами нужен отдельный совместимый bridge-release сайта, который сверит незавершённые операции с YooKassa и разрешит guard. Ручные установки на SQLite также не импортируются автоматически; этот менеджер обновляет только созданное им PostgreSQL-развёртывание.

Хранятся три последние версии сайта и менеджера. Резервные копии создаются ежедневно через systemd timer около 03:20 с случайной задержкой до 30 минут и также перед каждым обновлением или восстановлением базы. Восстановление выбранной базы сначала создаёт страховочный бэкап без запуска retention, закрывает внешний доступ и проверяет архив через validation-start без background jobs. При ошибке или прерывании до commit страховочная БД и прежний Nginx возвращаются из EXIT trap.

Архив конфигурации предназначен для аварийного ручного восстановления и содержит секреты. Автоматический пункт меню восстанавливает только PostgreSQL. Перед переносом бэкапов во внешнее хранилище используйте шифрование.

## YooKassa

Интеграцию можно пропустить при установке и включить позднее в `Environment settings -> YooKassa`. Менеджер запросит shop ID и secret key, сгенерирует отдельный webhook token и покажет URL вида:

```text
https://example.com/payments/yookassa/webhook?token=GENERATED_TOKEN
```

Этот URL необходимо зарегистрировать в кабинете YooKassa. `YOOKASSA_RETURN_URL` указывает на `/payment-return` и при смене домена обновляется автоматически, но webhook в кабинете YooKassa нужно изменить вручную.

## Размещение файлов

| Путь | Содержимое |
| --- | --- |
| `/opt/vpn-site/releases/<sha>` | Неизменяемые версии сайта и virtualenv |
| `/opt/vpn-site/current` | Активная версия сайта |
| `/opt/vpn-site-manager/releases/*` | Версии менеджера |
| `/opt/vpn-site-manager/current` | Активная версия менеджера |
| `/etc/vpn-site/vpn-site.env` | Секреты и настройки приложения, `root:vpn-site`, mode `0640` |
| `/etc/vpn-site/manager.conf` | Состояние менеджера, `root:vpn-site`, mode `0640` |
| `/var/backups/vpn-site/database` | Дампы PostgreSQL |
| `/var/backups/vpn-site/config` | Архивы конфигурации и копии env |
| `/etc/nginx/sites-available/vpn-site` | Управляемая конфигурация Nginx |
| `/etc/systemd/system/vpn-site*` | Сервис приложения и таймер бэкапов |

## Безопасность и ограничения

- Запускайте менеджер только от root; одновременные операции блокируются через `flock`.
- Env-файл содержит пароль БД, SMTP, Remnawave, YooKassa и ключ сессий. Не публикуйте его и не добавляйте в Git.
- Менеджер обновляет всю ОС через `apt-get full-upgrade`; если `/var/run/reboot-required` существует, перезагрузку нужно выполнить в согласованное окно самостоятельно.
- UFW управляется только когда вы явно включили его в мастере установки. Менеджер не удаляет сторонние правила.
- Скрипт проверяет наличие DNS-записи, но не может достоверно подтвердить изнутри сервера правильность NAT, маршрутизации и внешнего firewall.
- Полностью автоматического удаления сайта нет: это сознательно исключает случайное удаление production-данных.

## Разработка

Локальные проверки не изменяют систему:

```bash
bash tests/run.sh
```

Они проверяют синтаксис Bash, валидаторы, миграцию старого env, рендер Nginx, systemd-контракт и Python-инструменты env-файла. Если рядом расположен `vpn-site`, дополнительно сравниваются production env-схема и семантика Nginx обоих репозиториев. Полный интеграционный тест установки требует отдельной чистой VM с Ubuntu 24.04, рабочим DNS, PostgreSQL, systemd и тестовым приватным репозиторием.
